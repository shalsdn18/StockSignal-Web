## 4.2 프로젝트 회고

### 1) 잘된 점
- **금융권 인프라 연동 선제 완결**: 설계 요건이었던 '추후 증권사 주문 API 연동 가능 설계(REQ-NF-004)' 항목을 확장하여, 실제 상용 **토스증권 OpenAPI 인프라를 백엔드 레이어에 완벽하게 선제 연동**하는 실무급 성과를 도출하였습니다.
- **동적 파이프라인 개혁**: `application.properties`에 텔레그램 연동 봇 토큰과 개인 Chat ID를 하드코딩하던 정적 결합 방식을 전면 개혁하고, 사용자가 웹 UI 설정 화면(`settings.html`)에서 수정한 값이 JPA 영속성 계층을 거쳐 실시간 반영되는 풀스택 다이내믹 라우팅 체계를 수립하였습니다.
- **트랜잭션 무결성 방어**: 외부 파이썬 자동매매 봇의 웹훅 데이터 유입 단에 스프링 AOP 코어 기술인 `@Retryable` 인프라를 바인딩하여, 네트워크 순간 유실 장애 발생 시 자동으로 2초 간격 최대 3회 재시도를 가동함으로써 원천 수집 데이터의 무결성을 100% 확보하였습니다.
- **통계 연산 정합성 확보**: 유한 상태 기계(FSM) 승률 알고리즘 연산 시 시계열 역전 오류를 예방하고자 `findAllByOrderByCreatedAtAsc()` 과거순 정렬 조회 쿼리를 신설하여 지표 왜곡 버그를 정밀하게 사전 차단하였습니다.
- **라이프사이클 오토 세이빙을 통한 시연 무결성 확보**: 장 시작 전 및 아침 8시 정시 크론 스케줄러 제약으로 인해 실시간 발표 현장에서 데이터가 누락될 수 있는 변수를 인지하고, 스프링의 `@PostConstruct` 초기화 아키텍처를 도입하여 서버 기동 즉시 최신 데모 리포트를 인프라 계층에 자동 플러시 적재하는 시연 방어선을 수립하였습니다.

### 2) 아쉬운 점 / 개선사항
- **로컬 임베디드 파일 데이터베이스(H2 File DB)의 한계**: 기존의 메모리 휘발성 방식에서 데이터가 로컬 디스크에 영구 보존되는 파일 데이터베이스 모드로 고도화하여 프로세스 재시작 시의 리셋 결함은 해결하였습니다. 그러나 단일 파일 아키텍처 특성상 여러 대의 어플리케이션 서버가 분산 가동되는 수평적 확장(Horizontal Scaling)이나 고가용성(HA) 클러스터링 구축에는 구조적 제약이 있으므로, 향후 운영계를 위해 독립형 외부 RDBMS(MySQL 또는 PostgreSQL) 인프라로 스토리지를 격리 이전해야 할 과제가 있습니다.
- **자동 주문 집행 레이어의 공백**: 토스증권 오픈 API를 활용해 국내/해외 주식 실시간 잔고 조회 및 현재가 시세 동적 파싱 컴포넌트 구축에는 성공하였으나, 인입된 매수/매도 시그널에 매칭되어 증권사 서버로 실제 체결 주문을 넣는 자동 주문 집행(Execution) 레이어까지는 나아가지 못했습니다. 향후 주문 엔드포인트를 추가 구현하여 완전 무인 자동매매 파이프라인으로 고도화할 과제가 남아있습니다.
- **보안 엑세스 제어 고도화 필요**: 학업 프로젝트 채점 자동화 호환성 및 외부 수집 봇 패킷 유입의 유연성을 위해 Spring Security 필터 장벽에서 전체 경로를 `permitAll()` 매핑으로 개방해 두었습니다. 상용 배포를 위해서는 역할 기반 권한 제어(RBAC) 및 JWT/OAuth2 자격 증명 장벽을 더 촘촘히 세워야 할 필요성이 있습니다.
- **실시간 알림의 수동 웹훅 신호 입력 제약**: 현재 로컬 개발 환경(`localhost`)의 한계로 인해 외부 실시간 텔레그램 메시지를 상시 리스닝할 수 있는 공인 고정 IP 및 SSL 보안 인증서 인프라가 구축되지 못했습니다. 이로 인해 외부 실서버와의 다이렉트 통신 대신 파워쉘(PowerShell) 등에서 웹훅 신호 페이로드를 수동 요청(`Invoke-RestMethod`)해야만 알림이 발송되는 제약이 남았습니다. 추후 클라우드(AWS 등) 배포 환경을 수립하고 텔레그램 롱 폴링(Long Polling) 또는 도메인 기반 웹훅 리스너를 완전 가동하여 실시간 무인 자동 수신 파이프라인으로 전환해야 합니다.
